# Common Module Design Document

**Project:** Chemical Plant Digital Twin (CPDT)  
**Module:** common  
**Version:** 1.0  
**Status:** Completed  
**Author:** Lakshmi Sanjeev

The **Common** module serves as the shared foundation of the Chemical Plant Digital Twin (CPDT) platform. It contains reusable domain models, data transfer objects (DTOs), enumerations, and utility classes that are required by both the Simulator and Backend modules.

---

## 1. Module Objectives

The primary objectives of the Common module are:

- Provide a single source of truth for all shared domain models.
- Eliminate duplicate class definitions across modules.
- Ensure type-safe communication between system components.
- Standardize telemetry and alert message formats.
- Provide reusable utility classes used throughout the application.
- Improve maintainability through modular software design.

---

## 2. Module Responsibilities

The Common module is responsible for defining all shared structures used by the application.

Its responsibilities include:

- Defining immutable Data Transfer Objects (DTOs).
- Defining reusable domain models.
- Defining system-wide enumerations.
- Providing common utility classes.
- Supporting serialization of telemetry messages.
- Providing MQTT topic generation utilities.
- Providing standardized device identifier generation.

The module intentionally excludes:

- Business logic
- Alarm evaluation
- Digital Twin state management
- Device simulation
- Database persistence
- MQTT communication

## 4. Package Descriptions

### 4.1 dto

The `dto` package contains immutable Data Transfer Objects used for communication between the simulator and backend.

These objects define the structure of messages exchanged through MQTT without containing any business logic.

### 4.2 enums

The `enums` package defines all system-wide enumerations.

Enumerations provide standardized values throughout the application, preventing invalid states and improving code readability.

### 4.3 models

The `models` package defines the core domain objects shared by all modules.

These objects represent real-world entities within the chemical plant.

These models are independent of persistence frameworks and simulation logic.

### 4.4 utils

The `utils` package contains reusable helper classes that perform common operations required across the project.

Each utility class is stateless and designed for reuse.

## 5. Components

### Device

Represents a simulated industrial device within the chemical plant.

Each device stores identifying information, installation location, device type, process area, and operational metadata.

### Location

Represents the physical location of a device inside the plant.

This class enables spatial identification of assets without introducing application-specific logic.

### TelemetryPacket

Represents telemetry generated by simulated devices.

A telemetry packet contains:

- Device identifier
- Measurement type
- Measurement value
- Timestamp
- Process area

It serves as the primary communication format between the simulator and backend.

### AlertMessage

Represents alarm notifications generated by the backend.

It contains information including:

- Alert severity
- Device identifier
- Measurement information
- Timestamp
- Alarm state

### DeviceStatusMessage

Represents operational status updates for devices.

Examples include:

- ONLINE
- OFFLINE
- MAINTENANCE
- RETIRED

These messages enable synchronization of device status between modules.

### AlarmState

Defines the lifecycle state of an alarm.

Typical states include active, cleared, and acknowledged, allowing consistent alarm management throughout the application.

### AlertSeverity

Defines the severity level assigned to generated alerts.

Severity values distinguish between warning and critical conditions, enabling appropriate system responses.

### DeviceStatus

Defines the operational state of devices.

Standardized status values simplify monitoring and state transitions across the platform.

### DeviceType

Defines the categories of industrial devices represented in the simulation.

Using enumerations prevents inconsistent device classifications.

### MeasurementType

Defines the supported process variables measured within the plant.

Examples include:

- Temperature
- Pressure
- Humidity
- Gas Concentration
- Flow Rate
- Level
- pH

Each measurement type provides a standardized identifier for telemetry generation and processing.

### ProcessArea

Defines the different operational sections of the chemical plant.

Process areas provide logical grouping of devices and support area-specific monitoring and alarm evaluation.

### DeviceIdGenerator

Generates unique identifiers for simulated devices.

Centralizing identifier generation ensures consistent naming conventions across the application.

### TopicBuilder

Constructs standardized MQTT topic strings.

Centralizing topic creation prevents inconsistencies between publishers and subscribers.

### TelemetrySerializer

Converts telemetry objects into JSON format for MQTT transmission.

It encapsulates serialization logic, allowing messaging components to remain independent of JSON implementation details.

## 6. Module Dependencies

The Common module is intentionally independent of application-specific functionality.

It depends on:

- Jackson Databind

The common module has no dependency on the Backend module or the simulator module.

Instead, both application modules depend on the Common module. This dependency structure minimizes coupling and ensures that shared definitions remain reusable.

---

## 7. Design Decisions

- **Centralized Shared Definitions:** All shared classes are maintained in a single module to eliminate duplication and maintain consistency across the application.
- **Immutable DTOs:** Communication objects are implemented as immutable records wherever possible. Immutable data structures simplify concurrent processing and reduce unintended side effects.
- **Enumeration-Based Domain Modeling:** Finite system states are represented using enumerations instead of string constants.
- **Stateless Utility Classes:** Utility classes contain only reusable helper methods and maintain no internal state.
- **Separation of Concerns:** The Common module deliberately excludes business logic.
