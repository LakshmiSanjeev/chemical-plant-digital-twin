# Backend Module Design Document

**Project:** Chemical Plant Digital Twin (CPDT)  
**Module:** backend  
**Version:** 1.0  
**Status:** Completed  
**Author:** Lakshmi Sanjeev  

The **Backend** module serves as the Digital Twin layer of the Chemical Plant Digital Twin (CPDT) platform. It receives telemetry generated by the Simulator module, validates and persists incoming data, evaluates process alarms using configurable thresholds and hysteresis, maintains the real-time digital representation of plant devices, and exposes REST APIs for monitoring and visualization.

---

## 1. Module Objectives

The primary objectives of the Backend module are:

* Receive telemetry from the Simulator through MQTT.
* Validate incoming telemetry data.
* Persist device and telemetry information.
* Evaluate process alarms using configurable thresholds.
* Apply hysteresis to prevent alarm chattering.
* Maintain the real-time Digital Twin state of the plant.
* Publish device operational status updates.
* Expose REST APIs for accessing Digital Twin data.
* Serve as the central processing component of the CPDT platform.

---

## 2. Module Responsibilities

The Backend module is responsible for processing telemetry generated by the Simulator and maintaining the Digital Twin.

Its responsibilities include:

* Receiving telemetry through MQTT.
* Validating incoming telemetry packets.
* Persisting devices and telemetry data.
* Managing alarm threshold configuration.
* Evaluating warning and critical alarm conditions.
* Applying alarm hysteresis.
* Managing alarm state transitions.
* Persisting generated alerts.
* Updating device operational status.
* Maintaining Digital Twin state information.
* Publishing device status updates to the Simulator.
* Providing REST APIs for clients.

## 4. Package Descriptions

### 4.1 config

The `config` package contains configuration classes used by the Backend module.

It includes the MQTT subscriber configuration required for telemetry ingestion.

### 4.2 controllers

The `controllers` package exposes REST endpoints that provide access to Digital Twin information.

These controllers allow external clients to retrieve device information, telemetry, alerts, and current Digital Twin state.

### 4.3 entities

The `entities` package contains JPA entity classes representing database tables.

These entities provide persistence for devices, telemetry, alarm thresholds, and generated alerts.

### 4.4 exception

The `exception` package centralizes application exception handling.

It provides standardized error responses while isolating exception management from business logic.

### 4.5 mqtt

The `mqtt` package contains MQTT communication components.

It receives telemetry published by the Simulator and publishes device operational status updates back to the Simulator.

### 4.6 repositories

The `repositories` package provides Spring Data JPA repositories.

These repositories encapsulate database access while allowing service classes to remain independent of persistence implementation details.

### 4.7 services

The `services` package contains the business logic of the Backend module.

Services coordinate telemetry ingestion, persistence, alarm evaluation, alert generation, device management, and Digital Twin operations.

### 4.8 twin

The `twin` package maintains the real-time Digital Twin representation of the chemical plant.

It stores the latest state of every monitored device for rapid retrieval by REST APIs.

## 5. Components

### BackendApplication

Serves as the entry point of the Backend application.

It initializes the Spring Boot application and starts all Backend services.

### MqttIngestionConfig

Configures MQTT connectivity for telemetry ingestion.

It defines the MQTT subscriber responsible for receiving telemetry from the Simulator.

### AlertController

Provides REST endpoints for retrieving generated alerts.

It allows external applications to access alert history and monitoring information.

### DeviceController

Provides REST endpoints for retrieving device information maintained by the Backend.

### TelemetryController

Provides REST endpoints for accessing stored telemetry records.

### TwinController

Provides REST endpoints that expose the current Digital Twin state.

It enables external clients to retrieve live plant information.

### AlarmThresholdEntity

Represents alarm threshold records stored within the database.

Thresholds define warning and critical operating limits for each measurement type.

### AlertEntity

Represents persisted alert records generated during alarm evaluation.

Each alert stores severity, timestamp, measurement information, and associated device details.

### DeviceEntity

Represents industrial devices stored within the database.

It maintains operational metadata and current device status.

### TelemetryEntity

Represents telemetry records received from the Simulator.

Each entity stores measured values together with timestamps and associated device information.

### ErrorResponse

Defines the standardized structure of API error responses.

### GlobalExceptionHandler

Provides centralized exception handling across all REST controllers.

It converts application exceptions into consistent HTTP responses.

### InvalidTelemetryException

Represents invalid telemetry detected during validation.

### DeviceStatusPublisher

Publishes device operational status updates through MQTT.

These updates allow the Simulator to synchronize device states with the Digital Twin.

### TelemetryMessageHandler

Receives incoming MQTT telemetry messages.

It deserializes incoming messages and delegates processing to the telemetry ingestion service.

### AlarmThresholdRepository

Provides database access for alarm threshold records.

### AlertRepository

Provides persistence operations for generated alerts.

### DeviceRepository

Provides persistence operations for industrial device records.

### TelemetryRepository

Provides persistence operations for telemetry records.

### AlarmStateService

Maintains alarm state transitions for individual devices.

It applies hysteresis logic to prevent repeated alarm activation caused by fluctuating measurements.

### AlertEvaluationService

Evaluates incoming telemetry against configured alarm thresholds.

It determines warning and critical alarm conditions before generating alerts.

### AlertFetch

Provides retrieval operations for persisted alerts.

### AlertPersistenceService

Stores generated alerts within the database.

### DevicePersistenceService

Maintains device records and updates operational metadata when required.

### DeviceStatusService

Determines operational status changes for devices based on alarm conditions.

It publishes status updates whenever device state changes.

### TelemetryIngestionService

Coordinates the complete telemetry processing pipeline.

It validates incoming telemetry, updates devices, stores telemetry, evaluates alarms, updates the Digital Twin, and manages device status.

### TelemetryPersistenceService

Stores validated telemetry within the database.

### TelemetryQueryService

Provides retrieval operations for historical telemetry.

### TwinDeviceState

Represents the current Digital Twin state associated with a device.

It stores the latest telemetry values and operational information.

### TwinStateStore

Maintains the in-memory Digital Twin state for all devices.

It enables efficient retrieval of the latest plant state without repeatedly querying the database.

## 6. Backend Processing Workflow

Each telemetry message follows the processing sequence below:

1. The Simulator publishes telemetry through MQTT.
2. The Backend receives telemetry using the MQTT subscriber.
3. The telemetry message is deserialized.
4. Incoming telemetry is validated.
5. Device information is created or updated.
6. Telemetry is persisted to the database.
7. Alarm thresholds are retrieved.
8. Alarm evaluation is performed.
9. Alarm state and hysteresis logic are applied.
10. Generated alerts are persisted.
11. Device operational status is updated when required.
12. Updated device status is published back to the Simulator.
13. The Digital Twin state is refreshed.
14. REST APIs expose the latest plant information to clients.

This processing workflow ensures that the Backend continuously maintains an accurate Digital Twin representation of the simulated chemical plant.

---

## 7. Digital Twin State Management

The Backend maintains an in-memory Digital Twin that represents the latest operational state of every simulated device.

Each telemetry update refreshes the corresponding device state stored within the `TwinStateStore`.

Maintaining an in-memory representation provides several advantages:

* Rapid retrieval of current plant conditions.
* Reduced database access for frequently requested information.
* Efficient implementation of real-time monitoring dashboards.
* Simplified aggregation of device state by process area.

The Digital Twin remains synchronized with incoming telemetry throughout system operation.

## 9. Design Decisions

* **Layered Architecture:** Controllers, services, repositories, and entities are separated according to their responsibilities.
* **Digital Twin State Store:** An in-memory state store provides rapid access to the latest device information while complementing database persistence.
* **Configurable Alarm Thresholds:** Alarm limits are maintained independently of business logic, allowing threshold modification without code changes.
* **Hysteresis-Based Alarm Management:** Alarm hysteresis minimizes repeated alarm transitions caused by measurement fluctuations.
* **Repository Abstraction:** Database operations are isolated through Spring Data repositories.
* **Event-Driven Processing:** MQTT enables asynchronous telemetry ingestion and loose coupling between the Simulator and Backend.
* **Centralized Exception Handling:** Exception processing is centralized to improve maintainability and provide consistent API responses.
* **Separation of Concerns:** Communication, persistence, alarm evaluation, Digital Twin management, and API exposure are implemented as independent components.
