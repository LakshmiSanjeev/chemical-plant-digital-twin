# Simulator Module Design Document

**Project:** Chemical Plant Digital Twin (CPDT)  
**Module:** simulator  
**Version:** 1.0  
**Status:** Completed  
**Author:** Lakshmi Sanjeev

The **simulator** module serves as the simulation layer of the Chemical Plant Digital Twin (CPDT) platform. This package models the physical operation of a chemical plant by simulating industrial sensors, environmental conditions, operational scenarios, and telemetry generation. It serves as the source of real-time process data for the Digital Twin backend.

---

## 1. Module Objectives

The primary objectives of the Simulator module are:

- Simulate the operation of an industrial chemical plant.
- Generate realistic telemetry from virtual field devices.
- Model environmental conditions across different process areas.
- Simulate industrial sensor characteristics.
- Execute abnormal operating scenarios.
- Publish telemetry through MQTT.
- Provide configurable simulation cycles.
- Serve as a realistic data source for the Backend Digital Twin.

---

## 2. Module Responsibilities

The Simulator module is responsible for modeling the physical behavior of the virtual chemical plant.

Its responsibilities include:

- Initializing the virtual plant.
- Generating simulated industrial devices.
- Maintaining environmental conditions for each process area.
- Simulating industrial sensor measurements.
- Executing abnormal operating scenarios.
- Generating telemetry packets.
- Publishing telemetry through MQTT.
- Managing the overall simulation lifecycle.

## 4. Package Descriptions

### 4.1 bootstrap

The `bootstrap` package contains the application entry point used to start the simulator.

It initializes the simulation environment, creates the required components, and launches the simulation engine.

### 4.2 devices

The `devices` package contains concrete implementations of industrial sensors.

Each sensor measures a specific process variable while inheriting common sensing behavior from the abstract `SensorDevice` class.

### 4.3 engine

The `engine` package coordinates the execution of the simulator.

It manages simulation cycles, executes scenarios, and delegates telemetry publication through interchangeable publisher implementations.

### 4.4 environment

The `environment` package models the physical operating conditions of the chemical plant.

Environmental values stored within this package represent the actual state of each process area and are used by sensors to generate measurements.

### 4.5 mqtt

The `mqtt` package provides MQTT-specific communication components.

It contains the implementation responsible for publishing telemetry to the MQTT broker and subscribing to device status updates generated by the Backend module.

### 4.6 scenarios

The `scenarios` package models abnormal operating conditions within the plant.

Each scenario modifies one or more environmental variables to simulate realistic industrial events.

## 5. Components

### Test

Serves as the bootstrap class for the simulator.

It initializes the simulator and starts the execution of the simulation.

### SensorDevice

Represents the abstract base class for all simulated industrial sensors.

It provides common functionality shared by every sensor, including environmental sampling, measurement smoothing, calibration, measurement noise, ADC quantization, and drift simulation.

Concrete sensor implementations extend this class while defining measurement-specific behavior.

### Sensor Subclasses

Each device models the behavior of a specific industrial instrument, including its physical response characteristics, measurement algorithms,
noise sources, drift, and hardware limitations. Sensor implementations extend the `SensorDevice` abstraction and generate realistic
telemetry based on the current plant environment.

### PlantEnvironment

Represents the complete simulated chemical plant.

It maintains the environmental state for every process area and provides centralized access to plant conditions.

### ProcessAreaEnvironment

Represents the environmental conditions associated with a single process area.

It stores process variables such as temperature, pressure, gas concentration, flow rate, level, and pH that are continuously updated during simulation.

### Scenario

Defines the abstraction for abnormal operating scenarios.

Each scenario specifies how environmental conditions change over time while remaining independent of individual sensor implementations.

### FireScenario

Simulates fire-related incidents by increasing environmental temperature and hazardous gas concentration.

### GasLeakScenario

Simulates accidental gas leakage within the plant by increasing gas concentration in affected process areas.

### OverheatScenario

Simulates excessive operating temperatures resulting from abnormal process conditions.

### OverpressureScenario

Simulates elevated process pressure within industrial equipment.

### EquipmentFailureScenario

Simulates failures of industrial equipment that affect surrounding environmental conditions and sensor measurements.

### ScenarioEngine

Coordinates the execution of active scenarios.

It applies scenario effects to the plant environment during each simulation cycle.

### SimulatorEngine

Coordinates the execution of the complete simulator.

Its responsibilities include:

- Maintaining the simulation loop.
- Updating environmental conditions.
- Executing scenarios.
- Requesting sensor measurements.
- Generating telemetry.
- Delegating telemetry publication.

It serves as the central controller of the Simulator module.

### TelemetryPublisher

Defines the abstraction for publishing telemetry generated by the simulator.

This abstraction enables different publishing implementations without modifying the simulation engine.

### ConsoleTelemetryPublisher

Implements the `TelemetryPublisher` interface by writing telemetry to the console.

This implementation is primarily intended for debugging and development.

### MqttTelemetryPublisher

Publishes telemetry packets to MQTT topics for consumption by the Backend module.

It provides the production implementation of the `TelemetryPublisher` abstraction.

### DeviceStatusSubscriber

Subscribes to device status messages published by the Backend.

It receives operational status updates, such as maintenance or retirement notifications, allowing the simulator to remain synchronized with the Digital Twin.

## 6. Simulation Workflow

Each simulation cycle follows the sequence below:

1. The simulator initializes the plant environment.
2. The simulation engine begins the execution loop.
3. Active scenarios modify environmental conditions.
4. Sensors observe their corresponding environmental variables.
5. Sensor models apply filtering, calibration, drift, and measurement noise.
6. Telemetry packets are generated from sensor readings.
7. The selected telemetry publisher transmits the generated telemetry.
8. Device status updates received from the Backend are processed when available.
9. The simulation repeats after the configured execution interval.

This workflow continuously generates realistic telemetry for consumption by the Backend module.

## 7. Module Dependencies

The Simulator module depends on:

- Common module
- Eclipse Paho MQTT Client
- Jackson Databind
- SLF4J Logging

The Simulator module has no dependency on the Backend module.

Communication between the Simulator and Backend occurs exclusively through MQTT, ensuring loose coupling between both modules.

---

## 8. Design Decisions

- **Environment-Driven Simulation:** Environmental conditions are maintained independently of sensors. Sensors observe the environment rather than generating arbitrary values.
- **Abstract Sensor Hierarchy:** Common sensing behavior is centralized within the `SensorDevice` base class, reducing duplication across sensor implementations.
- **Scenario-Based Simulation:** Abnormal operating conditions are modeled as independent scenario classes, making new scenarios easy to introduce.
- **Publisher Abstraction:** Telemetry publication is separated from simulation logic through the `TelemetryPublisher` abstraction, allowing multiple publishing implementations.
- **Separation of Concerns:** Environment modeling, sensor simulation, scenario execution, and communication are implemented as separate components.
- **Loose Coupling:** Interaction with the Backend occurs only through MQTT, allowing both modules to evolve independently.
